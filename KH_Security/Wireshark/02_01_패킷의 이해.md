# Wireshark 패킷의 이해 (Ethernet, IP, ARP, ICMP)

## 개요
- Wireshark는 네트워크를 통해 전달되는 패킷을 캡쳐하고 분석하는 도구입니다.
- 패킷 분석을 위해서는 Ethernet Frame, IP Header, TCP/UDP Header 등의 구조를 이해해야 합니다.

---

## 대표적인 프로토콜 헤더 크기

- TCP Header : 20 byte  
- UDP Header : 8 byte  
- IP Header : 20 byte

- 패킷은 다음과 같은 계층 구조로 이루어진다.
```text
Application Layer  
Transport Layer (TCP / UDP)  
Network Layer (IP / ARP / ICMP)  
Link Layer (Ethernet)  
```

---

## Ethernet Frame 구조

- Ethernet Frame은 실제 네트워크에서 데이터가 전달되는 기본 프레임 구조이다.

### 구조

| 구분 | 설명 | 예시 |
|---|---|---|
| Preamble (8 byte) | 프레임의 시작을 알리고 송수신 장비 간 동기화를 맞추는 필드 (Wireshark에서는 캡쳐되지 않음) | `10101010` 패턴 |
| Destination MAC Address (6 byte) | 목적지 장비의 MAC 주소 | `00:1A:2B:3C:4D:5E` |
| Source MAC Address (6 byte) | 패킷을 전송한 장비의 MAC 주소 | `00:0C:29:AB:CD:EF` |
| Type (2 byte) | 상위 계층 프로토콜 종류를 나타내는 필드 | `0x0800 = IPv4`, `0x0806 = ARP`, `0x86DD = IPv6` |
| Data (Datagram) | 상위 계층 데이터(IP Datagram)가 들어가는 영역 | `IP Header + TCP Header + Data` |
| FCS (4 byte) | Frame Check Sequence, 프레임 오류 검출을 위한 값 (대부분 Wireshark에서 캡쳐되지 않음) | `CRC32 값` |

## IP Header 구조

![IP Header](/KH_Security/Wireshark/img/IP%20헤더.png)

| 필드 | 크기 | 설명 | 예시 |
|-----|-----|-----|-----|
| Version | 4 bit | IP 프로토콜 버전 | IPv4 = 4 |
| Header Length | 4 bit | IP Header의 길이 | 5 → 5 × 4byte = 20byte |
| Type Of Service | 8 bit | 서비스 우선순위 및 QoS 정보 | 일반 패킷 = 00000000 |
| Total Length | 16 bit | IP Datagram 전체 길이 (IP Header + Data) | Header 20B + Data 40B = 60B |
| Identification | 16 bit | Fragment된 패킷을 재조립하기 위한 식별 값 | Fragment된 패킷들이 동일한 ID 사용 (ex: ID=100) |
| Flags | 3 bit | Fragment 여부를 나타내는 필드 | DF=1 (Fragment 금지), MF=1 (뒤에 Fragment 존재) |
| Fragment Offset | 13 bit | Fragment된 데이터의 위치 | 첫 번째 Fragment = 0 |
| TTL | 8 bit | Time To Live, 라우터를 지날 때마다 1씩 감소 | 초기 TTL = 64 → Router 통과 후 63 |
| Protocol | 8 bit | 상위 프로토콜 종류 | 1=ICMP, 6=TCP, 17=UDP |
| Header Checksum | 16 bit | IP Header 오류 검출 | Header 손상 여부 확인 |
| Source IP Address | 32 bit | 출발지 IP 주소 | 192.168.0.10 |
| Destination IP Address | 32 bit | 목적지 IP 주소 | 8.8.8.8 |

### 패킷 분석 실습

![IP](/KH_Security/Wireshark/img/09.png)

- Ethernet
```
dst MAC (6byte) : 00 26 b6 d5 e1 08
src MAC (6byte) : c4 54 44 3a b8 81
type (2byte) : 08 00 → IP
```

- IP Header
```
version (4bit) : 0100 → IPv4
header length (4bit) : 0101 → 5 word → 20byte
TOS (1byte) : 00
total length (2byte) : 00 34 → 52byte
identification (2byte) : 75 0a → 29962
flag (3bit) : 010 → fragment 금지
fragment offset (13bit) : 0000000000000 → 0byte
TTL (1byte) : 80 → 128
protocol (1byte) : 06 → TCP
header checksum (2byte) : 00 00
src IP (4byte) : c0 a8 0a 09 → 192.168.10.9
dst IP (4byte) : 7d d1 e6 ee → 125.209.230.238
```

## ARP (Address Resolution Protocol)

- ARP는 IP 주소를 MAC 주소로 변환하는 프로토콜이다.

### ARP 동작 과정
```
- 송신 Host가 목적지 IP로 통신하려고 한다.
- ARP Broadcast를 네트워크에 전송한다.
- 해당 IP를 가진 Host가 ARP Reply를 전송한다.
- 송신 Host는 MAC 주소를 ARP Cache에 저장한다.
- 이후 통신은 해당 MAC 주소를 이용하여 수행된다.
```

---

## ARP Header 구조

![ARP Header](/KH_Security/Wireshark/img/ARP%20헤더.png)

| 필드 | 크기 | 설명 | 예시 |
|-----|-----|-----|-----|
| Hardware Type | 2 byte | 네트워크 타입 | Ethernet = 1 |
| Protocol Type | 2 byte | 사용되는 상위 프로토콜 | IPv4 = 0x0800 |
| Hardware Length | 1 byte | MAC 주소 길이 | Ethernet = 6 byte |
| Protocol Length | 1 byte | IP 주소 길이 | IPv4 = 4 byte |
| OP (Operation) | 2 byte | ARP 요청 또는 응답 구분 | 1 = ARP Request, 2 = ARP Reply |
| Sender MAC Address | 6 byte | ARP 요청을 보내는 장치의 MAC 주소 | 00:26:b6:d5:e1:08 |
| Sender IP Address | 4 byte | ARP 요청을 보내는 장치의 IP 주소 | 192.168.10.9 |
| Destination MAC Address | 6 byte | 목적지 장치의 MAC 주소 (요청 시 보통 00:00:00:00:00:00) | 00:00:00:00:00:00 |
| Destination IP Address | 4 byte | MAC 주소를 알아내려는 목적지 IP 주소 | 192.168.10.1 |

### 패킷 분석 실습

![ARP](/KH_Security/Wireshark/img/08.png)

- Ethernet
```
dst MAC (6byte) : 70 5d cc 04 be b0
src MAC (6byte) : 2c fd a1 72 a8 d4
type (2byte) : 08 06 → ARP
```

- ARP Header
```
hardware type (2byte) : 00 01 → Ethernet
protocol type (2byte) : 08 00 → IPv4
hardware size (1byte) : 06 → MAC 주소 길이 (6byte)
protocol size (1byte) : 04 → IP 주소 길이 (4byte)
opcode (2byte) : 00 01 → ARP Request

sender MAC address (6byte) : 2c fd a1 72 a8 d4
sender IP address (4byte) : c0 a8 0a 03 → 192.168.10.3

destination MAC address (6byte) : 70 5d cc 04 be b0
destination IP address (4byte) : c0 a8 0a 01 → 192.168.10.1
```

## ICMP (Internet Control Message Protocol)

- ICMP는 네트워크 통신 과정에서 발생하는 오류나 상태 정보를 전달하기 위한 프로토콜입니다.  
송신자와 목적지 노드의 네트워크 상태를 진단하는 역할을 한다.
- ICMP 메시지는 **IP Datagram 내부에 포함되어 전송**됩니다.

### ICMP - PING 구조

- Ping은 ICMP의 **Echo Request / Echo Reply 메시지**를 이용하여 통신 상태를 확인하는 방식입니다.

### 주요 ICMP Type

| Type | Code | 설명 |
|-----|-----|-----|
| 0 | 0 | Echo Reply (Ping 응답) |
| 8 | 0 | Echo Request (Ping 요청) |
| 3 | 0 | Network Unreachable |
| 3 | 1 | Host Unreachable |
| 5 | - | Redirect |
| 9 | 0 | Router Advertisement |
| 10 | 0 | Router Solicitation |

![ICMP 헤더](/KH_Security/Wireshark/img/ICMP%20헤더.png)

| 필드 | 크기 | 설명 | 예시 |
|-----|-----|-----|-----|
| Type | 1 byte | ICMP 메시지 종류 | 8 = Echo Request, 0 = Echo Reply |
| Code | 1 byte | Type에 대한 세부 코드 | Echo Request = 0 |
| Checksum | 2 byte | ICMP 메시지 오류 검증 | 0x3a4f |
| Identifier | 2 byte | Ping 프로세스를 식별하는 ID | 0x1c46 |
| Sequence Number | 2 byte | Ping 패킷 순서 번호 | 1, 2, 3 ... |
| Data / Option | 가변 | 추가 데이터 영역 | Ping payload |

### ICMP 패킷 분석

![10](/KH_Security/Wireshark/img/10.png)

- Ethernet
```
dst MAC (6byte) : 00 0c 29 81 f7 0f
src MAC (6byte) : 00 0c 29 84 bd f7
type (2byte) : 08 00 → IPv4
```

- IP Header
```
version (4bit) : 0100 → 4:IPv4)
header length (4bit) : 0101 → 5 word
TOS (1byte) : 00
total length (2byte) : 00 a6 → 166 byte
identification (2byte) : 00 00
flag (3bit) : 010 → Don't Fragment
fragment offset (13bit) : 0000000000000 → 0
TTL (1byte) : 40 → 64
protocol (1byte) : 01 → ICMP
header checksum (2byte) : a2 c6
src IP (4byte) : c0 a8 0b 11 → 192.168.11.17
dst IP (4byte) : c0 a8 0b 2f → 192.168.11.47
```

- ICMP Header
```
type (1byte) : 0b → 11 (Time Exceeded)
code (1byte) : 00 → TTL Exceeded
checksum (2byte) : 9e 14
```

- **패킷의 TTL이 0이 되어 라우터가 패킷을 폐기하면서 생성한 ICMP 메시지입니다.**

## ICMP Redirect

- ICMP Redirect는 **라우터가 더 효율적인 경로를 알려줄 때 사용하는 메시지**입니다.

### 동작 과정

- 호스트는 기본 게이트웨이(라우터 A)로 패킷을 전송합니다.
- 라우터 A는 라우팅 테이블을 확인하고 라우터 B가 더 효율적인 경로라고 판단합니다.
- 라우터 A는 패킷을 라우터 B로 전달합니다.
- 동시에 호스트에게 **ICMP Redirect 메시지**를 보내 앞으로는 라우터 B를 사용하도록 알려줍니다.
- 이후 호스트는 동일 목적지 패킷을 라우터 B로 직접 전송합니다.

### ICMP Redirect 확인해야 하는 IP

- Redirect 메시지에서는 다음 3개의 IP를 확인해야 합니다.
  - Redirect 원인이 된 목적지 IP
  - Redirect 메시지를 보낸 라우터 IP
  - 새롭게 사용해야 할 게이트웨이 IP

---

|-----|-----|-----|-----|
| Type | 1 byte | ICMP 메시지 타입 | 5 = Redirect |

| Destination IP (DI) | 4 byte | 패킷 목적지 IP | 8.8.8.8 |
| Data | 8 byte | 원본 데이터 일부 | Payload 일부 |

### ICMP 패킷 분석

![11](/KH_Security/Wireshark/img/11.png)

- Ethernet
```
dst MAC (6byte) : 4c ed fb bd 5d f1
src MAC (6byte) : 00 0c 29 35 e3 90
type (2byte) : 08 00 → IP
```

- IP Header
```
version (4bit) : 0100 → 4:IPv4
header length (4bit) : 0101 → 5 word
TOS (1byte) : 00
total length (2byte) : 00 38 → 56 byte
identification (2byte) : 33 72 → 13,170
flag (3bit) : 000 → 단편화 없음
fragment offset (13bit) : 0000000000000
TTL (1byte) : fe → 254
protocol (1byte) : 01 → ICMP
header checksum (2byte) : f3 fd → 1111 0011 1111 1101
src IP (4byte) : c0 a8 0a 01 → 192.168.10.1
dst IP (4byte) : c0 a8 0a 03 → 192.168.10.3
```

- ICMP Header
```
type (1byte) : 05 → ICMP Redirect
code (1byte) : 01 → Host Redirect
checksum (2byte) : af ff → 1010 1111 1111 1111
gateway address (4byte) : c0 a8 0a 40 → 192.168.10.64
```

- ICMP Header 내부 IP Header
```
version (4bit) : 0100 → (4:IPv4)
header length (4bit) : 5 word
TOS (1byte) : 00
total length (2byte) : 00 34 → 52 byte
identification (2byte) : a2 2f
flag (3bit) : 000 → fragmentation 가능
fragment offset (13bit) : 0000000000000 → 0
TTL (1byte) : fe → 254
protocol (1byte) : 06 → TCP
header checksum (2byte) : df e9 → 1101 1111 1110 1001
src IP (4byte) : c0 a8 0a 03 → 192.168.10.3
dst IP (4byte) : cb f8 fc 02 → 203.248.252.2
```

- 이 IP 헤더는 **ICMP Redirect 메시지 안에 포함된 원래 패킷의 IP Header**입니다.    
즉 **192.168.10.3 호스트가 203.248.252.2로 TCP 패킷을 보내는 과정에서 Redirect가 발생했다는 의미입니다.**

- 설명
```
본 패킷은 ICMP Redirect(Type 5, Code 1) 패킷이다.
송신지 IP는 192.168.10.1이고 수신지 IP는 192.168.10.3이다.
이 패킷은 192.168.10.3이 203.248.252.2로 전송하던 TCP 패킷에 대해,
향후 해당 목적지로의 통신 시 새로운 게이트웨이 192.168.10.64를 사용하도록 유도합니다.

따라서 패킷 분석상 Redirect 메시지를 보낸 주체는 192.168.10.1이며,
공격 관점에서는 192.168.10.3의 라우팅 경로를 192.168.10.64로 변경시키려는 시도로 해석할 수 있습니다.
```
