# Wireshark 패킷의 이해 (Transport Layer)

## 개요

- Transport Layer는 Host 간 데이터 전달을 담당하는 계층이다.  
- Network Layer(IP)가 패킷을 목적지까지 전달하는 역할을 한다면,  
Transport Layer는 데이터 전송의 신뢰성, 흐름 제어, 오류 검출 등의 기능을 수행합니다.

- 주요 기능
  - 데이터 흐름 제어 (Flow Control)
  - 오류 검출 (Checksum)
  - 연결 설정 및 해제
  - 데이터 재전송
  - 데이터 순서 보장

- 대표 프로토콜
  - TCP
  - UDP

---

## TCP (Transmission Control Protocol)

- TCP는 신뢰성 있는 데이터 전송을 제공하는 연결지향 프로토콜이다.

- 특징
  - 연결지향 방식
  - 신뢰성 있는 데이터 전달
  - 데이터 순서 보장
  - 흐름 제어
  - 재전송 기능 제공
 
### TCP Connection Initialization (Three-Way Handshake)

- TCP는 데이터를 전송하기 전에 송신자(sender)와 수신자(receiver) 사이에 connection을 먼저 설정합니다.  
연결이 생성된 이후에 data segment 전송이 시작됩니다.

- TCP 연결이 초기화될 때 다음 정보들이 설정됩니다.
  - Sequence Number (Seq#)
    - 데이터의 순서를 관리하기 위한 번호입니다.
    - 각 호스트는 자신의 초기 sequence number를 설정합니다.
  - Buffer : `네트워크에서 받은 데이터를 임시로 저장하는 메모리 공간입니다.`
    - 수신된 데이터를 임시로 저장하는 공간입니다.
    - 데이터가 순서대로 도착하지 않을 경우 버퍼에 저장됩니다.
  - Flow Control 정보 : `Receiver의 버퍼 상태를 기준으로 Sender의 전송 속도를 조절하여 데이터 손실을 방지하는 메커니즘입니다.`
    - 수신된 데이터를 임시로 저장하는 공간입니다.
    - 데이터가 순서대로 도착하지 않을 경우 버퍼에 저장됩니다.

#### Three-Way Handshake

- TCP 연결은 3단계 과정으로 생성됩니다.

##### Step 1
```
Client → Server (SYN segment 전송)

- client가 server에게 연결 요청합니다.
- client의 초기 sequence number 설정합니다.
- 데이터는 포함되지 않습니다.
```

##### Step 2
```
Server → Client (SYN + ACK 전송)

- server가 SYN 요청을 수신
- client 요청 승인
- server의 초기 sequence number 설정
- buffer 및 연결 변수 할당
```

##### Step 3
```
Client → Server (ACK 전송)

- client가 SYNACK를 수신하고 ACK 응답
- client의 buffer 및 연결 변수 할당
- 이후 데이터 전송 가능
```

#### Closing a Connection

- TCP 연결을 종료할 때는 데이터를 보내는 방향이 서로 독립적이기 때문에 양쪽에서 각각 종료 절차를 수행해야 합니다.  
따라서 TCP Closed Connection는 총 4단계 과정으로 이루어지게 됩니다.

![Closing Connection](/KH_Security/Wireshark/img/TCP%20Closing%20Connection.png)

##### Step 1
```
- Client → Server (FIN 전송)
```

- 설명
```
- Client는 connection을 종료하기 위해 FIN bit가 1로 설정된 TCP segment를 Server에게 전송합니다.
- Client는 더 이상 보낼 데이터가 없다는 것을 알리는 것입니다.
```

##### Step 2
```
Server → Client (ACK 전송, FIN 전송)
```

- 설명
```
- Server는 FIN을 수신하면 먼저 ACK로 응답합니다.  
  이후 Server도 자신의 연결 종료를 위해 FIN segment를 Client에게 전송합다.
```

##### Step 3
```
Client → Server (ACK 전송)
```

- 설명
```
- Client는 Server의 FIN을 수신하면 ACK로 응답합니다.  
  이후 일정 시간 동안 대기 상태(Time Wait)를 유지한 후 connection을 종료합니다.
- Time Wait 상태는 마지막 ACK가 정상적으로 전달되었는지 확인하기 위해 존재한다.
```

##### Step 4
```
Server connection 종료
```

- 설명
```
- Server는 Client의 ACK를 수신하면 connection을 종료합니다.
```

### TCP Header 구조

![TCP 헤더](/KH_Security/Wireshark/img/TCP%20헤더.png)

| 필드 | 크기 | 설명 |
|-----|-----|-----|
| Source Port | 2 byte | 송신 애플리케이션 포트 |
| Destination Port | 2 byte | 목적지 애플리케이션 포트 |
| Sequence Number | 4 byte | 전송 데이터 시작 번호 |
| Acknowledgment Number | 4 byte | 다음에 받을 데이터 번호 |
| Header Length | 4 bit | TCP 헤더 길이 |
| Reserved | 6 bit | 예약 필드 |
| Flags | 6 bit | TCP 제어 플래그 |
| Window Size | 2 byte | 수신 가능한 데이터 크기 |
| Checksum | 2 byte | 오류 검출 |
| Urgent Pointer | 2 byte | 긴급 데이터 위치 |
| Options | variable | 추가 옵션 |

### TCP Flags

- TCP는 연결 제어를 위해 여러 플래그를 사용한다.

| Flag | 설명 |
|-----|-----|
| SYN | 연결 요청 |
| ACK | 응답 확인 |
| FIN | 연결 종료 |
| RST | 연결 재설정 |
| PSH | 데이터를 즉시 전달 |
| URG | 긴급 데이터 표시 |

- Flag 자세한 설명
```
URG (1) : urgent point에 있는 데이터를 읽어도 됩니다.
URG (0) : urgent point에 있는 데이터를 무시합니다.

SYNC : 클라이언트가 서버한테 통신하자고 패킷을 보내는 것입니다.
클라이언트가 먼저 보냅니다. 서버는 이제 sync와 ack를 보냅니다. 

ACK (1) : ack # 데이터가 의미가 있다.
ACK (0) : ack # 데이터가 의미가 없다. - 0인데 ack 데이터가 있으면 무시합니다.

PUSH (1) : sender가 리시버에게 보낸다 - 버퍼링 하지 않았으면 좋겠다는 의미입니다.(어플리케이션에 바로 주라는 뜻) 
- 순서에 맞는 패킷이 오지 않아서 버퍼링을 하는 것인데 왜 하는가? 
  - push에서 버퍼링 하지 말라는 건 큐잉지연을 최소화 해달라는 것입니다.

RESET : 패킷이 500번까지 보냈는데 갑자기 ack 1300번이 받는다면
버퍼에 문제가 있다고 생각하여 시퀀스 넘버를 초기화 하자 이런 의미로 reset 패킷을 보냅니다.

FIN : 접속을 끊자고 패킷을 보내는 것
- 클라이언트가 보내는데 서버가 보내는 경우도 있습니다.
(idle time 아무것도 안하고 클라이언트가 가만히 있으면 어느 타임에 끊어주세요라는 뜻이다. 세션이 너무 길어지는 것을 막아준다.)
```

## UDP (User Datagram Protocol)

- UDP(User Datagram Protocol)는 전송 계층에서 사용되는 프로토콜로 TCP와 달리  
**비연결 지향(Connectionless) 방식으로 데이터를 전송하는 프로토콜**입니다.  

- UDP는 상대방과 연결을 설정하지 않고 데이터를 바로 전송하며, 송신 시스템이 전송한 데이터에 대해 **수신 시스템의 응답을 확인하지 않습니다.**    
즉, 데이터 전송에 대한 **확인 절차가 생략된 프로토콜**이다.

- 이러한 특징 때문에 프로토콜 동작이 단순하며 **네트워크 부하를 줄일 수 있다는 장점**이 있지만    
데이터 전달 과정에서 오류 검증이나 재전송 절차가 없기 때문에 **데이터의 신뢰성을 보장하지 못합니다.**

- 특징
  - 연결 설정 과정 없음
  - 빠른 데이터 전송
  - 신뢰성 보장 없음
  - 재전송 기능 없음
  - 패킷 손실 가능

---

### UDP Header 구조

- udp 헤더가 짧은 이유 : UDP는 연결 관리나 신뢰성 기능이 없기 때문에 헤더가 매우 단순하고 짧습니다.

![UDP](/KH_Security/Wireshark/img/UDP%20헤더.png)

| 필드 | 크기 | 설명 |
|-----|-----|-----|
| Source Port | 2 byte | 송신 포트 |
| Destination Port | 2 byte | 목적지 포트 |
| Length | 2 byte | UDP 데이터그램 전체 길이 |
| Checksum | 2 byte | 오류 검출 |

- 추가 설명
```
- Source Port
송신 호스트에서 데이터를 보낸 응용 프로그램의 포트 번호를 의미합니다. 
응답이 필요한 경우 수신 호스트가 이 포트 번호를 이용해 송신 프로그램으로 데이터를 다시 보냅다.

- Destination Port
데이터를 수신할 응용 프로그램의 포트 번호입니다.  
수신 호스트는 이 포트 번호를 통해 어떤 애플리케이션이 데이터를 처리해야 하는지 결정합니다.

- UDP Datagram Length
UDP 데이터그램의 전체 길이를 나타냅니다.
길이 범위 : 0 ~ 65,535 byte
- 즉, UDP 패킷의 전체 크기를 나타내는 필드입니다.

- Checksum
Checksum 필드는 데이터 오류 검출을 위한 값입니다.  
송신 측에서 계산된 체크섬 값이 패킷에 포함되며, 수신 측에서는 동일한 계산을 수행하여 값이 동일한지 확인합니다.
만약 값이 다르면 데이터가 전송 중 손상된 것으로 판단하게 되지만 UDP는 TCP와 달리 오류가 발생해도 재전송을 수행하지 않습니다.
즉, 오류 검출만 수행하고 복구는 하지 않습니다.
```

## 정리

Transport Layer는 Host 간 데이터 전달의 신뢰성과 흐름을 관리하는 계층이며, TCP는 신뢰성 중심의 프로토콜이고 UDP는 속도 중심의 프로토콜이다.
